# Custom Thread Pool

## Описание проекта
В рамках данной работы реализован собственный Thread Pool без использования стандартного ThreadPoolExecutor. Основная цель - продемонстрировать понимание механизмов управления потоками, очередями задач, балансировки нагрузки и обработки перегрузок.
Пул поддерживает динамическое масштабирование, несколько очередей задач, балансировку нагрузки и настраиваемую политику отказа.

## Основные возможности
### Настраиваемые параметры
* corePoolSize — минимальное количество потоков
* maxPoolSize — максимальное количество потоков
* queueSize — размер очереди задач
* keepAliveTime — время жизни неактивного потока
* minSpareThreads — минимальное количество свободных потоков

### Распределение задач
Используется алгоритм `Round Robin`, это обеспечивает равномерное распределение нагрузки и простоту реализации:
* каждая задача направляется в очередь следующего воркера
* очереди распределены между потоками (1 очередь = 1 воркер)

### Управление потоками

* При старте создаются `corePoolSize` потоков
* Если число свободных потоков меньше `minSpareThreads`, создаются дополнительные потоки
* Потоки завершаются при простое (длительность простоя = `keepAliveTime`), если их больше `corePoolSize`


### Обработка отказов

Все очереди заполнены, достигнут `maxPoolSize`.

Задача отклоняется с подробной диагностикой:

* `[Pool] (Rejected) Task Task-#id was rejected due to overload!`
* `(total rejected: N, workers: M/maxPoolSize, reason: all queues full and maxPoolSize reached)`

Счётчик `total rejected` и текущее число воркеров позволяют в режиме реального времени оценивать интенсивность перегрузок.

#### Выбранный подход и его недостатки:
Простое отклонение с логированием. Достоинство — предсказуемость, задача не теряется молча. Недостаток — вызывающий поток не получает результата; альтернативные стратегии (CallerRuns, DiscardOldest) могут быть предпочтительнее в зависимости от требований.

### Логирование

Система выводит сообщения по следующим ключевым событиям:

* создание потоков
* завершение потоков
* добавление задач в очередь
* выполнение задач
* idle timeout
* отказ в обработке задач

Пример:

`[Pool] Task accepted into queue #1: Task-5`; `[Worker] MyPool-worker-2 executes Task-5`; `[Worker] MyPool-worker-2 idle timeout, stopping.`


## Архитектура
### Основные компоненты:

* CustomThreadPool — управление потоками и задачами
* Worker — выполнение задач
* CustomBlockingQueue — очередь задач с идентификатором
* CustomThreadFactory — создание потоков


## Анализ производительности

### Сравнение с ThreadPoolExecutor

| Критерий           | Custom Pool | ThreadPoolExecutor |
| ------------------ | ----------- | ------------------ |
| Гибкость           | высокая     | высокая            |
| Производительность | средняя     | высокая            |
| Надежность         | средняя     | высокая            |
| Простота           | ниже        | выше               |

Вывод: стандартный `ThreadPoolExecutor` более оптимизирован и устойчив, однако собственная реализация позволяет гибко управлять поведением системы.

## Влияние параметров на производительность

### corePoolSize

* слишком маленький → задержки
* слишком большой → лишние ресурсы

### maxPoolSize

* влияет на обработку пиковых нагрузок

### queueSize

* маленький → больше отказов
* большой → выше задержка

### minSpareThreads

* уменьшает задержку при всплесках нагрузки
* увеличивает потребление ресурсов


## Балансировка нагрузки

Используется алгоритм `Round Robin`:

```
Task → Queue 0 → Queue 1 → Queue 2 → ...
```

Преимущества:

* простота
* равномерность

Недостатки:

* не учитывает текущую загрузку очередей

Возможное улучшение:

* алгоритм `Least Loaded` (выбор наименее загруженной очереди)

## Демонстрация работы

В классе `Main` реализованы следующие сценарии (каждый использует отдельный пул с параметрами):

1. Normal Load — 8 задач при ёмкости 16: все принимаются, очереди не заполнены, воркеры работают без давления.
2. Full Load — 14 задач при ёмкости 16: все принимаются, очереди заполнены полностью, воркеры работают на максимуме.
3. Overload (Rejections) — 25 задач при ёмкости 16: 9 задач отклоняются с логированием.
4. Forced Shutdown — 8 долгих задач (10 сек), через 1 секунду вызывается `shutdownNow()`: все выполняющиеся задачи прерываются.
5. Graceful Shutdown — 8 задач, затем `shutdown()`: все задачи из очередей завершаются, задача поданная после `shutdown()` отклоняется.


## Вывод

В ходе работы был реализован собственный пул потоков с поддержкой:

* динамического управления потоками
* балансировки задач
* обработки перегрузки
* логирования

Проект демонстрирует принципы работы многопоточных систем и служит упрощённой моделью серверных решений.
